# AVI-OS Skill — Coding Quality Standards

## Scope

This skill applies to ALL code generation, review, and modification tasks. No exceptions.

## Mandatory Standards

### ISO Alignment

- **ISO/IEC 25010**: All code assessable against 8 quality characteristics — functional suitability, reliability, performance efficiency, operability, security, compatibility, maintainability, portability.
- **ISO/IEC 5055**: Zero critical weaknesses in security, reliability, performance, maintainability. Measure via static analysis (CWE-based).

### OWASP Top 10:2021 Compliance

All code must defend against: broken access control, security misconfiguration, supply chain failures, vulnerable components, injection, cryptographic failures, authentication failures, integrity failures, logging failures, and mishandled exceptions.

### SOLID Principles (Mandatory)

- Single Responsibility: one class/module = one reason to change
- Open/Closed: extend via abstraction, never modify working code
- Liskov Substitution: subtypes fully substitutable for base types
- Interface Segregation: no client depends on unused methods
- Dependency Inversion: depend on abstractions, never concretions

### Clean Code Rules (Mandatory)

- Meaningful, intention-revealing names (no abbreviations, no single letters except loop counters)
- Functions ≤ 30 lines, do one thing, one level of abstraction
- Files ≤ 400 lines
- Parameters ≤ 3 per function
- Nesting ≤ 3 levels deep
- Cyclomatic complexity ≤ 10 per function
- Zero dead code, zero commented-out code
- DRY: no duplicated logic; extract shared code
- KISS: simplest solution that works
- YAGNI: no speculative features
- Comments explain "why", never "what"
- Boy Scout Rule: leave code cleaner than found

### TypeScript Specific (Avi's Stack)

- Strict mode: strict, noImplicitAny, strictNullChecks, strictFunctionTypes
- No `any` types unless explicitly justified with a comment
- Use enums, union types, or branded types over raw strings/numbers
- Use readonly where mutation is not required
- Prefer functional patterns: map/filter/reduce over imperative loops
- All async operations must have error handling (try/catch or .catch)
- Use DTOs with class-validator for all API input
- Use Guards/Interceptors/Pipes per NestJS conventions

### Security by Default

- Parameterised queries only — never concatenate user input into SQL
- Input validation on all external data (API requests, file uploads, environment variables)
- Output encoding for all rendered content
- Least privilege: minimum necessary permissions for every service/user
- Secrets in Key Vault / environment variables — zero hardcoded credentials
- Dependencies scanned for CVEs — zero critical/high vulnerabilities
- HTTPS only, TLS 1.2+, strong cipher suites
- Authentication: JWT with short-lived tokens + refresh rotation
- CORS: explicit allowlist, never wildcard in production

### Error Handling

- All errors caught and handled — no unhandled promise rejections
- Fail fast, fail loud — detect problems at the boundary, not deep in the stack
- Structured logging with correlation IDs (request ID propagated through all layers)
- Operational errors (expected) vs programmer errors (bugs) — handle differently
- Graceful shutdown: drain connections, finish in-flight requests, then exit
- Never expose stack traces or internal details to API consumers

### Testing Requirements

- Minimum 80% line coverage, 70% branch coverage for all new code
- Unit tests: test business logic in isolation, mock external dependencies
- Integration tests: test module interactions with real (or containerised) dependencies
- No tests that depend on execution order
- Test names describe the scenario and expected outcome
- Edge cases tested: null, undefined, empty string, boundary values, error paths

### Infrastructure as Code

- All infrastructure through Terraform — zero manual portal changes
- Plan before apply — always review plan output
- Remote state with locking
- Modules for reusability, variables for configuration
- Tagging strategy: Environment, Project, ManagedBy on all resources
- Docker: multi-stage builds, non-root user, .dockerignore, health checks, no secrets in images

### Code Review Checklist (Every PR)

Before any code is merged, verify:

1. SOLID principles followed?
2. Security: no injection, no hardcoded secrets, input validated?
3. Error handling: all paths covered, structured logging?
4. Tests: coverage targets met, edge cases included?
5. Naming: clear, consistent, intention-revealing?
6. Performance: no N+1 queries, no unbounded loops, resource limits set?
7. Documentation: public APIs documented, complex logic explained?
8. Dependencies: no new critical/high CVEs introduced?

## Enforcement

If code generated by Claude does not meet these standards, Claude must flag the gap explicitly and remediate before delivering. Quality is non-negotiable — no shortcuts for speed.
